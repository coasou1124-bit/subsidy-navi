# 暮らしAIアシスタント — サービス設計書

> Version 1.0 / 2026-06-28

---

## 1. サービスコンセプト

**「知らなくて損した」をゼロにする、暮らしのAI相談窓口。**

日本には国・都道府県・市区町村を合わせると数千種類の公的支援制度が存在する。しかし多くの市民は「自分が対象かどうかわからない」「申請の手間が怖い」「そもそも制度の存在を知らない」という理由で、本来受け取れるはずの給付を見落としている。

本サービスは「検索サイト」ではなく「AI相談員」として機能する。ユーザーの状況をヒアリングし、「なぜあなたが対象になるか」を人間の言葉で説明し、申請まで伴走することで、給付の受給率を高めることを目的とする。

## 2. ターゲットユーザー

### プライマリ

| セグメント | 代表的な状況 | 主な悩み |
|---|---|---|
| 子育て世帯 | 0〜18歳の子を持つ親 | 育児給付・保育費・教育費の支援を知りたい |
| 就職・転職者 | 離職中・求職活動中 | 雇用保険・職業訓練・生活支援を調べたい |
| 介護者 | 親の介護が始まった30〜50代 | 介護保険・ヘルパー費用の制度を把握したい |
| 低所得・生活困窮 | 収入が急減したひとり親世帯など | 生活保護・家賃支援・食料支援を今すぐ知りたい |

### セカンダリ

- 地方自治体の窓口職員（情報を住民に正確に伝えたい）
- 社会福祉士・FP・弁護士（クライアントの支援制度を調査したい）

## 3. 解決する課題

### 現状の3つの壁

**壁①「発見できない」**
- 制度名が難解で検索キーワードが思いつかない
- 国・県・市の制度が縦割りで、横断検索ができない

**壁②「判断できない」**
- 対象条件が複雑で自分が当てはまるかわからない
- 専門用語が多く、内容を理解できない

**壁③「申請できない」**
- 必要書類・窓口・期限が分散していて手続きが重い
- 「申請しないと受け取れない」制度が多い（不申請損失）

### 本サービスの解決アプローチ

```
発見 → 対話型ヒアリングで自動検出
判断 → 「なぜあなたが対象か」を個別理由で説明
申請 → 必要書類・時間・難易度・期限を1枚で提示
```

## 4. 他サービスとの差別化

| 比較軸 | 政府の給付金検索 | 一般検索エンジン | **本サービス** |
|---|---|---|---|
| 個人最適化 | △ 条件入力あるが単純 | ✗ キーワード検索のみ | ◎ 対話で詳細プロフィール取得 |
| 理由の説明 | ✗ 制度一覧を返すのみ | ✗ 記事を返すのみ | ◎「なぜあなたが対象か」を明示 |
| 見落とし提案 | ✗ | ✗ | ◎ 関連制度を能動的に提案 |
| 申請サポート | △ リンクのみ | ✗ | ◎ 難易度・時間・書類を整理 |
| 自治体連携 | ◎（公式） | ✗ | △ → β以降に順次対応 |
| 会話の継続性 | ✗ | ✗ | ◎ ライフイベントをまたいだ提案 |

## 7. データ設計

### 7-1. 制度データ（Subsidy）

```typescript
interface Subsidy {
  id: string                    // 例: "national-001"
  name: string                  // 制度名
  category: SubsidyCategory     // childcare / housing / employment / ...
  providerLevel: 'national' | 'prefectural' | 'municipal'
  prefecture?: string           // 都道府県（国制度は省略）
  city?: string                 // 市区町村
  officialUrl: string           // 公式ページURL
  conditions: {
    ageMin?: number
    ageMax?: number
    hasChildren?: boolean
    employmentTypes?: EmploymentType[]
    incomeMax?: string          // 所得上限の目安
    familyStructures?: FamilyStructure[]
  }
  aiSummary: {
    amount: string              // 受取金額の目安
    targetPeople: string        // 対象者の説明
    deadline: string            // 申請期限（文章）
    deadlineDate?: string       // ISO日付（残日数計算用）
    requiredDocuments: string[]
    applicationPlace: string    // 申請窓口
    notes: string[]
    difficulty: number          // 1〜5
    estimatedApplicationTime: string
  }
  priority: number              // 検索優先度（高いほど上位）
  updatedAt: string             // データ更新日（ISO）
}
```

### 7-2. ユーザープロフィール（UserProfile）

```typescript
interface UserProfile {
  age: number
  prefecture: string
  city: string
  familyStructure: FamilyStructure
  hasChildren: boolean
  childrenCount: number
  childrenAges?: number[]       // β〜 詳細な子どもの年齢
  employment: EmploymentType
  incomeRange: IncomeRange
  concerns: SubsidyCategory[]
  hasDisability?: boolean       // β〜
  isCaregiver?: boolean         // β〜 介護中かどうか
}
```

### 7-3. カテゴリ定義

```typescript
type SubsidyCategory =
  | 'childcare'   // 子育て
  | 'education'   // 教育
  | 'housing'     // 住宅
  | 'employment'  // 雇用
  | 'medical'     // 医療
  | 'income'      // 生活・収入
  | 'elderly'     // 介護・老後
  | 'disability'  // 障がい
  | 'other'
```

### 7-4. 制度データの管理方針（MVP期間）

- 国の制度（12件〜）: `data/subsidies/national.ts` に静的データとして管理
- 都道府県制度（β〜）: `data/subsidies/prefectural/[pref].ts` に分割管理
- 市区町村制度（正式版〜）: DBまたはCMS管理に移行

## 8. AIの役割と設計方針

### 現在のAI（ルールベース）

チャットフローは `lib/chatFlow.ts` のステートマシンで実装。  
`filterSubsidies()` の `scoreSubsidy()` 関数がルールに基づいてスコアリングを行い、`matchReasons` を生成する。

```
入力: UserProfile
処理: 条件マッチング → スコアリング → 理由生成
出力: SearchResult[] + missedSubsidies[] + nextConsultations[]
```

### β版以降のAI強化方針

| フェーズ | AI活用 | 技術 |
|---|---|---|
| MVP | ルールベース（現在） | 静的ロジック |
| β | 自由質問への回答 | Claude API（claude-haiku-4-5） |
| β | 個別理由文の自然化 | Claude API でmatchReasonsを自然文に変換 |
| 正式版 | 申請書のひな型生成 | Claude API + ドキュメント生成 |
| 正式版 | 制度データの自動更新検知 | Claude API で行政サイトの変更を要約 |

### AIを使う際の原則

1. **ファクトはルールベース**で担保。AIは「説明の自然化」にのみ使う
2. 制度の適否判断はAIではなく、構造化条件でチェックする（幻覚リスク排除）
3. 回答の末尾に必ず「公式サイトでご確認ください」の免責を付与

## 9. 自治体制度データの取得方法

### 段階的なデータ収集戦略

#### MVP期: 手動管理（国の制度のみ）

- 厚生労働省・こども家庭庁・国土交通省などの公式サイトから主要制度を手動で収集
- 更新頻度が低い（年1〜2回）ため、手動で `data/subsidies/national.ts` を更新

#### β期: 半自動収集（都道府県レベル）

| 手法 | 対象 | 頻度 |
|---|---|---|
| e-Gov API | 法令・告示ベースの国制度 | 月次クロール |
| 各都道府県の制度一覧ページスクレイピング | 県単位の制度 | 四半期ごと |
| 「すまいの給付金」「子育て支援」等の公式ポータル | 特定分野 | 随時 |

#### 正式版: API連携・データ提携

| 手法 | 概要 |
|---|---|
| J-LIS（地方公共団体情報システム機構）連携 | 自治体データの一元アクセス |
| LoGoフォーム / LG-WAN連携 | 自治体の内部システムとの接続 |
| 自治体との直接契約（B2B） | 制度データを自治体が自ら登録・更新 |

### データ鮮度の管理

- 各制度に `updatedAt` フィールドを持たせ、古いデータをUI上で「要確認」表示
- ユーザーが「この情報は古いようです」と報告できるフラグ機能（β〜）

| 対象 | 料金感 | 提供内容 |
