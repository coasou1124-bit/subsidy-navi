## 【1】データ取得方法の比較

### 比較マトリクス

| 方法 | コスト | 精度 | 鮮度 | スケール | 法的リスク |
|---|---|---|---|---|---|
| 手動登録 | 高（人件費） | ◎ | △ | × | ◎ |
| 自治体オープンデータ | 低 | ○ | △ | △ | ◎ |
| 行政API | 低 | ◎ | ◎ | ○ | ◎ |
| RSS | 低 | △ | ◎ | △ | ○ |
| スクレイピング | 中 | △ | ○ | ○ | △ |
| AI自律検索 | 中 | △ | ○ | ◎ | △ |

---

### 1-A. 手動登録

**概要**: 運営者または委託スタッフが公式情報を確認しながら直接入力する。

**メリット**
- 最も正確。誤情報を通すかどうかを人間が判断できる
- 公式サイトの文脈（注意事項・例外規定）を正しく読み取れる
- 法的リスクが最小（著作権問題なし）

**デメリット**
- スケールしない（全国1700以上の自治体に対応不可）
- 更新漏れが起きやすい（担当者の負荷・見落とし）
- 人件費が継続的にかかる

**MVP段階での採用判断**: 全国共通制度30〜100件は手動登録でスタート。品質担保の基盤として継続。

---

### 1-B. 自治体オープンデータ

**概要**: 総務省が推進する「オープンデータ」として自治体が公開するCSV・JSON・XMLを取得する。

**主要データソース**
| 提供元 | URL | 形式 | 内容 |
|---|---|---|---|
| e-Stat（政府統計） | estat.go.jp | API/CSV | 統計・制度の件数データ |
| 社会保障・税番号制度 | digital.go.jp | API | マイナンバー関連制度 |
| 各省庁データカタログ | data.e-gov.go.jp | CSV/JSON | 法令・行政手続き |
| 子ども家庭庁 | cfa.go.jp | PDF/HTML | 子育て制度 |

**メリット**
- 利用規約が明確（クリエイティブ・コモンズ CC BY 4.0が多い）
- 構造化されているためパース容易
- 無料

**デメリット**
- 制度「一覧」はあっても、申請方法・必要書類などの詳細データがない
- 自治体ごとにフォーマットがバラバラ（標準化は進行中だが未完）
- 更新タイミングが年1回程度の自治体も多い

**法的注意点**
- 「政府標準利用規約（第2.0版）」が適用される場合、商用利用・改変・再配布が可能
- ただし「出典の明示」が必要（「出典：○○省」を画面に表示する）

---

### 1-C. 行政API

**概要**: 国・自治体が公式に提供するAPIを叩いてデータを取得する。

**現状で利用可能なAPI**
| API名 | 提供元 | 利用可能な情報 | 費用 |
|---|---|---|---|
| e-Gov 法令API | デジタル庁 | 法令条文（制度の根拠法） | 無料 |
| e-Gov 電子申請API | デジタル庁 | 申請様式・手続き一覧 | 無料 |
| eLTAX API | 地方税機構 | 地方税関連手続き | 登録制・無料 |
| 年金情報API | 将来検討中 | 年金見込み額 | 未提供 |

**メリット**
- 最も信頼性が高い（一次情報）
- 利用規約が明確
- 定期的な更新保証

**デメリット**
- 現状では制度の「申請方法・金額・条件」レベルのAPIはほぼ存在しない
- APIが整備されているのは法令条文レベルに留まる
- 整備が進んでいるのはデジタル庁管轄のみ

**将来展望**: デジタル行政推進により2027〜2028年頃には制度情報APIが整備される可能性あり。今から連携する設計にしておく価値はある。

---

### 1-D. RSS

**概要**: 省庁・自治体のRSSフィードを購読し、制度の新設・変更を検知する。

**主要RSSソース**
| 提供元 | フィード内容 |
|---|---|
| 厚生労働省 | 報道発表資料（制度改正を含む） |
| こども家庭庁 | 子育て関連施策の新着情報 |
| 総務省 | 地方行政関連情報 |
| 文部科学省 | 教育・奨学金関連 |
| 各都道府県公式サイト | 制度の新設・変更情報 |

**メリット**
- リアルタイムに近い変更検知が可能
- 公式情報なので著作権問題なし（タイトル・URLのみ取得）
- 実装コストが低い

**デメリット**
- RSSに含まれるのはニュース見出しのみ。「何がどう変わったか」の詳細は別途確認が必要
- 全自治体がRSSを提供しているわけではない
- 誤検知が多い（制度改正と関係ないプレスリリースも流れる）

**採用方法**: 「変更があったかもしれない制度のアラート」として使う。RSS検知 → 担当者に通知 → 手動確認 というフローで運用。

---

### 1-E. スクレイピング

**概要**: 行政・自治体の公式サイトをクローリングし、制度情報を自動抽出する。

**技術的アプローチ**
```
[対象URL] → [HTML取得] → [パース・抽出] → [差分検出] → [DB更新候補] → [人間確認] → [公開]
```

**メリット**
- APIがない自治体の情報も取得できる
- スケールする（プログラムで1700自治体を網羅可能）
- 更新サイクルを自由に設定できる

**デメリット**
- サイト構造変更で突然壊れる（メンテナンスコストが高い）
- 抽出精度が低い（表組みやPDFは特に難しい）
- 自治体ごとにパーサーを書く必要がある

**⚠️ 法的注意点（重要）**

| 観点 | 詳細 |
|---|---|
| **利用規約確認** | 各サイトの利用規約に「クローリング禁止」条項がないか確認必須 |
| **robots.txt** | `robots.txt` の `Disallow` を必ず尊重する |
| **著作権法** | 行政の「法令・判決・告示」は著作権法13条により著作物に該当しないが、行政機関が独自に作成した説明文は保護対象になりうる |
| **不正アクセス禁止法** | ログイン必須ページへのアクセスは違反 |
| **サーバー負荷** | 過度なアクセスは業務妨害になりうる。クロール間隔は最低10秒以上 |

**現実的な運用**
- 国・都道府県レベル: `robots.txt` を確認したうえでスクレイピング可能なサイトのみ対象
- 市区町村レベル: 全量は困難。主要都市（人口50万以上）から段階的に対応
- PDF掲載が多い自治体: 手動登録に切り替える

---

### 1-F. AI自律検索

**概要**: AIに「○○市の子育て支援制度を検索して情報をまとめて」と指示し、Web検索結果から構造化データを抽出する。

**技術**: Claude / GPT-4o のWeb Search機能、または Perplexity API

**メリット**
- 非構造化テキストからの情報抽出が得意
- PDFや複雑なHTMLも処理できる
- スケールが比較的容易

**デメリット**
- ハルシネーション（誤情報生成）のリスクがある。金額・期限・条件の誤りは致命的
- API利用コストがかかる（1件あたり数円〜数十円）
- 生成内容の著作権帰属が不明確（法整備が進行中）

**採用方法**: 「下書き生成」として活用し、必ず人間が確認してから公開する。AI単独で自動公開するのはリスクが高い。

---

### 結論：ハイブリッド戦略

```
┌─────────────────────────────────────────────────┐
│ 情報源の優先順位                                    │
│                                                 │
│  ① 行政API          （最優先・存在する場合）           │
│  ② 公式RSSで変更検知 （アラートとして活用）              │
│  ③ 手動登録          （精度担保の基盤）               │
│  ④ スクレイピング    （スケールのため・確認前提）         │
│  ⑤ AI補助           （下書き・翻訳・要約のみ）          │
└─────────────────────────────────────────────────┘
```

**原則**: AIとスクレイピングは「下書き生成」まで。最終的な公開判断は人間が行う。

## 【2】更新フロー

### 制度変更検知から公開までの標準フロー

```
【トリガー発生】
    │
    ├─① RSS/スクレイパーが変更を検知（自動）
    ├─② 担当者が公式サイトで確認（手動）
    └─③ ユーザーから「情報が古い」報告（フォーム）
    │
    ▼
【ステータス: 要確認】
    担当者にSlack/メール通知
    admin_logs に action:'change_detected' を記録
    │
    ▼
【担当者が公式サイトと照合】
    │
    ├─ 変更なし → last_verified_at を更新 → 完了
    │
    └─ 変更あり
           │
           ▼
       【管理画面で編集】
           ・変更内容を入力
           ・変更前の値は履歴として admin_logs に保存
           ・ステータス: 下書き
           │
           ▼
       【AIサマリ再生成（必要な場合）】
           ・金額・条件・期限が変わった場合のみ
           ・AIが新しい要約を下書き生成
           ・担当者が確認・修正
           │
           ▼
       【公開前チェックリスト】
           □ 金額が公式サイトと一致
           □ 申請期限が正しい
           □ 公式URLが有効（404でない）
           □ 免責事項の更新が必要か確認
           │
           ▼
       【ステータス: 公開】
           updated_at を更新
           admin_logs に action:'updated', changed_fields を記録
           │
           ▼
       【ユーザーへの影響確認】
           ・条件変更の場合: 既存フィルタリングロジックへの影響確認
           ・終了の場合: is_active=false で非表示
```

### 制度終了フロー

```
【制度が終了した場合】
    │
    ▼
【管理画面で「終了」操作】
    ・is_active = false
    ・end_date = 終了日
    ・admin_logs に action:'deactivated'
    │
    ▼
【表示の扱い】
    ・検索結果・チャット結果から除外
    ・詳細ページは残す（'この制度は終了しました' バナー表示）
    ・SEOインデックスはそのまま（ユーザーが検索で来た場合に案内できる）
```

### 緊急修正フロー（金額・期限の誤りが発覚した場合）

```
【誤情報発覚】
    │
    ▼
【即時: is_active = false で一時非表示】（5分以内）
    │
    ▼
【修正内容を確認・入力】（1時間以内）
    │
    ▼
【再公開】
    admin_logs に action:'emergency_fix', notes:'誤情報修正' を記録
```

### 3-D. AI要約更新

「AIで再生成する」ボタンを押すと:

```
1. 公式URL のページ内容を取得
2. 現在の条件情報（age, category, etc.）をコンテキストとして渡す
3. 以下プロンプトでAIが下書きを生成:
   「以下の情報を元に、以下の形式でサマリを作成してください。
    金額・期限・条件の事実のみを抽出し、想像で補わないこと。
    判断できない場合は空欄にすること。」
4. 生成結果を「下書き」としてフォームに表示
5. 担当者が確認・修正してから保存
```

**AIが単独で公開することは禁止。必ず人間が最終確認。**

### Phase 1 — 全国共通制度（2026年7〜9月）

**目標**: 国が定める主要制度を100件、高品質で揃える

**対象省庁**
| 省庁 | 主な制度 | 件数目標 |
|---|---|---|
| こども家庭庁 | 児童手当・育児給付・保育無償化 | 15件 |
| 厚生労働省 | 雇用保険・傷病手当・高額療養費 | 30件 |
| 文部科学省 | 奨学金・就学支援 | 10件 |
| 国土交通省 | 住宅ローン減税・省エネ補助 | 10件 |
| 内閣府・総務省 | 生活保護・障がい年金 | 15件 |
| 日本年金機構 | 遺族年金・老齢年金 | 10件 |
| その他 | 難病助成・特定医療費 | 10件 |

**品質基準**
- 公式URLが有効であること
- 金額が数値または根拠ある記述であること
- 申請方法が具体的に記載されていること
- 最終確認日が30日以内であること

**担当体制**: 運営者1〜2名で月50件ペースで追加・確認

---

- 人間によるサンプル確認フローが確立されている

**自動化の対象と限界**

| 自動化できること | 自動化しないこと |

  → 担当者が承認 → 公開

## 【5】法的・運営面

### 5-A. 行政情報の著作権

**著作権法 第13条（権利の目的とならない著作物）**
```
次の各号のいずれかに該当する著作物は、この章の規定による権利の目的となることができない。
一  憲法その他の法令
三  国若しくは地方公共団体の機関、独立行政法人...が発する
    告示、訓令、通達その他これらに類するもの
```

**つまり**:
| 情報の種類 | 著作権 | 利用可否 |
|---|---|---|
| 法令条文（育児・介護休業法など） | なし | ◎ 自由に利用可 |
| 告示・通達（支給額の基準告示等） | なし | ◎ 自由に利用可 |
| 省庁のウェブサイト上の解説文 | **あり** | △ 要注意 |
| 各省庁パンフレットの文章 | **あり** | △ 要注意 |

**安全な対応**:
- 法令条文・告示はそのまま引用可能（出典明示）
- ウェブサイトの文章は「参考にAIが要約する」形式にする
- 「〇〇省資料を元に暮らしAIアシスタントが要約」と明示する

---

### 5-B. スクレイピングの法的リスク

**法律ごとの整理**

| 法律 | リスク | 対策 |
|---|---|---|
| 著作権法 | 全文コピー・転載は侵害の可能性 | 「要約・情報の構造化」にとどめる |
| 不正競争防止法 | データベース権侵害の可能性 | 構造化データの「全量コピー」を避ける |
| 不正アクセス禁止法 | 認証突破は違反 | ログイン不要のページのみ対象 |
| 電子計算機損壊等業務妨害罪 | 過度なクロールでサーバーに負荷 | クロール間隔を10秒以上に設定 |
| 利用規約違反 | 民事上のリスク | robots.txt + 利用規約を確認 |

**robots.txt チェックの実装例**
```typescript
// スクレイパー実行前に必ず確認
async function canScrape(url: string): Promise<boolean> {
  const { hostname } = new URL(url)
  const robotsUrl = `https://${hostname}/robots.txt`
  const res = await fetch(robotsUrl)
  const text = await res.text()
  // robots.txt パーサーで確認
  return !isDisallowed(text, url)
}
```

**現実的なスタンス**:
行政機関の多くは「情報の普及・活用」を目的としており、robots.txt で明示的に禁止していない限り、適切な頻度での情報収集は問題になりにくい。ただし:
- 個々の自治体のIT担当が厳格な場合もあるため、問い合わせを推奨
- 「利用目的が公共の利益」である旨をUser-Agentに明示する
  `User-Agent: KurashiAI-Bot (+https://kurashi-ai.jp/bot-info)`

### 5-D. 更新責任と運営体制

**更新責任のマトリクス**

| 制度の種類 | 更新頻度 | 確認担当 | 確認方法 |
|---|---|---|---|
| 国の主要制度 | 年2回（4月・10月） | 運営者（専任） | 公式サイト直接確認 |
| 補正予算・臨時給付 | 随時 | 運営者（専任） | RSS監視 + 即時対応 |
| 都道府県制度 | 年1回（4月） | 担当スタッフ | チェックリスト運用 |
| 市区町村制度 | 年1回（4月） | 担当スタッフ | 自動確認 + 抽出 |

**「最終確認日」の表示**

ユーザーに対して情報の鮮度を明示する:

```
この制度情報は 2026年6月28日 に公式サイトと照合済みです。
申請前は必ず公式サイトでご確認ください。
```

これにより:
- ユーザーが「古い情報かもしれない」と自ら判断できる
- 「確認した」という事実を記録として残せる
- 法的な免責の根拠にもなる

### 5-E. 将来の規制リスクと対策

| リスク | 発生条件 | 対策 |
|---|---|---|
| 行政機関からの警告 | 誤情報で申請者が損害を受けた場合 | 即時非公開 + 謝罪ページ表示の仕組みを事前整備 |
| 著作権侵害申告 | 全文コピー・パンフ転載が発覚した場合 | 「AI要約」形式を徹底。転載はしない |
| 個人情報漏洩 | ユーザーの相談内容をログ保存した場合 | チャットログをサーバーに保存しない |
| 薬機法・医療広告規制 | 「この制度で病気が治る」等の表現 | 医療制度は金額・申請方法のみ案内。効果の言及は禁止 |
